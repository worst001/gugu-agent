# Gateway 负载与韧性优化计划

## 背景

当前 gateway 已经承载登录、订单、支付回调、Gugu Managed 模型转发、GLM 文件解析和官网页面。短期重点不是马上上复杂集群，而是先把“服务器不会被异常请求打爆”做好，再按流量逐步拆分。

最近检查到的服务形态：

- 服务器规格：约 2 核 CPU、3.5 GB 内存、40 GB 系统盘。
- Gateway 进程：Bun 服务，常驻内存约 90 MB。
- 暴露方式：Bun 监听本机端口，Nginx 对外提供 443，并保留 8787 兼容代理。
- 高风险入口：AI 对话转发、GLM 文件解析、支付回调、下载页/购买页流量。

## 当前承载能力估算

在现有单机、SQLite、本机 Bun 进程架构下，建议按保守值设计：

- 注册或在线设备：几千台问题不大，设备本身不等于并发请求。
- 同时活跃 AI 请求：50-100 个是当前更稳的目标区间。
- 同时 GLM 文件解析：建议限制在 5-10 个，短期不要放开到几十个。
- 试运营日活：100-300 DAU 较稳；300-800 DAU 要看附件解析比例；1000+ DAU 需要先做限流、监控和队列。

真正的瓶颈通常不是页面访问，而是外部模型转发占用的长连接、附件上传/解析、数据库写锁和异常重试风暴。

## 立即要做的保护

1. 请求体限制
   - Nginx 和应用层都限制 body 大小。
   - 普通聊天 JSON 严格限制。
   - GLM 文件解析按格式区分：图片更小，文档最大 20 MB，压缩包不走远程解析。

2. 并发限制
   - AI 对话转发设置全局并发上限。
   - GLM 文件解析设置更低的独立并发上限。
   - 单设备、单 IP、单套餐都要有速率限制。

3. 超时与熔断
   - 上游模型请求必须有连接超时、首包超时和总超时。
   - GLM 连续失败时短时间熔断，直接提示稍后重试。
   - 支付回调必须快速返回，耗时任务不要阻塞回调线程。

4. 积分与成本保护
   - 先校验文件类型、大小、套餐、余额，再调用上游模型。
   - 上游失败不应错误扣点。
   - 重复请求需要幂等键，避免用户重复点击造成多次扣点或多次发码。

## 数据与文件策略

短期可以继续 SQLite，但要补强：

- 开启 WAL 和合理的 `busy_timeout`。
- 订单、激活码、支付回调、扣点流水必须有幂等字段。
- 每日自动备份数据库到 OSS。
- 备份文件至少保留 7-30 天。
- 支付回调原始摘要可以保留，敏感内容不要长期明文保存。

附件策略：

- 普通压缩包优先本地解压到用户项目，不上传 gateway。
- 不支持本地解压的压缩包直接提示用户手动解压，不走 GLM。
- 远程解析只允许白名单格式和大小内文件。
- 未来如必须上传，应走 OSS 临时对象、短生命周期、自动清理，而不是长期落在 gateway 磁盘。

## 监控与预警

最低限度要有这些指标：

- CPU 连续 5 分钟超过 80%。
- 可用内存低于 500 MB。
- 磁盘使用率超过 80%。
- Nginx 5xx 比例超过 1%。
- Gateway p95 响应时间超过 3 秒。
- AI 对话 p95 超过 90 秒。
- GLM 解析 p95 超过 120 秒。
- GLM 并发超过 10。
- 上游模型错误率超过 5%。
- SQLite busy/locked 错误出现明显增长。
- 备份失败。

推荐先做一个轻量方案：Nginx access/error log + gateway 结构化日志 + 每分钟健康检查脚本 + 异常时企业微信/飞书通知。

## 扩容路线

### Phase 1：单机加固

- 加 Nginx 限流和 body 限制。
- 应用层加并发池、队列、超时、熔断。
- SQLite WAL、备份和慢查询日志。
- Dashboard 增加关键指标：当前并发、上游错误率、今日 token、今日附件解析数。

### Phase 2：存储与任务拆分

- 将长耗时 GLM 解析放入任务队列。
- 主请求只创建任务，前端轮询任务状态。
- 文件临时对象放 OSS，设置生命周期自动删除。
- 支付和发码保持强幂等。

### Phase 3：数据库升级

- 从 SQLite 迁移到 PostgreSQL 或 MySQL。
- 订单、设备、扣点流水、支付回调用正式索引。
- 增加只读报表查询，避免 dashboard 扫主表影响线上。

### Phase 4：多实例部署

- Gateway 做无状态化。
- Nginx 或 SLB 挂多个 gateway 实例。
- Session、队列、限流状态放 Redis。
- 灰度发布和回滚脚本标准化。

## Docker 建议

Docker 值得做，但它不是第一优先的扩容手段。它的价值主要是：

- 部署一致，减少“本地能跑、服务器不行”。
- 方便回滚镜像。
- 方便后期多实例扩容。
- 可以把 gateway、nginx、备份任务、监控脚本拆成清晰服务。

建议等 Phase 1 的限流、超时、备份和日志补齐后，再做 Docker 化。否则只是把不稳定的进程装进容器，核心风险仍然存在。

## 优先级

1. 严格限制 Gugu Managed/GLM 上传格式、大小、并发。
2. Nginx + 应用层双重限流。
3. SQLite WAL、幂等、备份。
4. 结构化日志和预警。
5. GLM 解析任务队列。
6. Docker 化和多实例。

这条路线能让试运营阶段先稳住成本和可用性，再逐步过渡到更标准的生产架构。
