# 如何降低 DeepSeek V4 Agent 的缺点

## 问题背景

使用 Claude Code Agent 跑 DeepSeek V4 时，虽然成本更低、速度可能更快，但在复杂工程任务中可能出现以下问题：

- 复杂规划稳定性较弱
- 多步骤任务容易偏离目标
- 工具调用可靠性取决于适配层
- 长上下文和仓库级理解不如顶级模型稳定
- 返工成本可能抵消低价优势

核心思路不是让 DeepSeek V4 独立承担所有工作，而是把它放在更适合的位置：**执行明确、边界清楚、可验证的任务**。

## 解决原则

让 DeepSeek V4 做：

- 有明确方案的代码修改
- 低风险批量任务
- 可自动测试验证的任务
- 单文件或小范围改动
- 按 checklist 执行的任务

不要让它独立负责：

- 架构设计
- 大型重构
- 跨模块复杂规划
- 隐含业务规则判断
- 无明确验收标准的开放式任务

## 推荐工作流

```text
1. 强模型 Plan：分析需求、确定方案、识别风险
2. DeepSeek V4 执行：按明确方案做小范围修改
3. 自动验证：运行 lint、test、build
4. DeepSeek V4 修复简单失败
5. 强模型 Review：检查 diff、兼容性、边界条件
6. 必要时再让 DeepSeek V4 按 review 修改

----------------------------------

目标：
实现 `具体目标`。

范围：
- 只允许优先修改：`文件/目录`
- 不要修改无关模块
- 不要做无关重构

执行流程：
1. 先列出你准备修改的文件和原因
2. 再做最小实现
3. 修改后运行指定测试
4. 如果测试失败，只修与本任务相关的问题

约束：
- 保持现有 API/行为兼容
- 遵循项目现有代码风格
- 不引入新的架构抽象，除非必要
- 如果需要修改超过 5 个文件，先暂停并说明原因

验证方式：
- 运行：`npm test`
- 运行：`npm run lint`
- 无法运行时，说明原因并给出手动验证步骤

输出要求：
- 改了什么
- 为什么这样改
- 如何验证
- 是否存在风险
